These are the API docs for the OpenBEL Framework.

Style

Indentation
  • Four spaces should be used as the unit of indentation.
  • Indent with spaces, not tabs.
  • Indentation should be consistent.

Line Length

  • Avoid lines longer than 80 characters.

Brace Usage

  • Conditionals should always use braces.
  • Braces in loops can be considered optional, e.g.,
    for (int i = 0; i < array.length; methodcall(array[i]), i++);
Switch Statements
  • Switch statements should always include default cases.

Blank Usage

  • Keywords followed by parantheses
  • Commas in arguments lists
  • Expressions in for loops
  • Binary operators, except '.'
  • After casts

E.g.,
  • if (...) { /// }
  • method(a, b, c);
  • for (int i = 0; i < j; j++) { // }
  • list.add((T) item);

Documentation

  • Document all public, default and protected elements
  • Document all thrown exceptions
  • Use the tags and tools provided (e.g., use of inheritDoc)
  • Document when nulls are accepted or returned for a method

Examples, note the use of may be language: 
        
          /**
           * Creates an item.
           *
           * @param e Entry, which may be null
           */
          public Item(Entry e) {
              this.entry = e;
          }

          /**
           * Returns the item's entry.
           *
           * @return Entry, which may be null
           */
          public Entry getEntry() {
              return entry;
          }

          /**
           * Returns the item's other entries.
           *
           * @return {@link Set} of entries, which may be empty
          public Set getEntries() {
              if (entries == null) return emptySet();
              return entries;
          }

Conventions

Catching Exceptions

Avoid catching higher-level exceptions; there are seldom reasons to do so. 
            
              try {
                  // some code
              } catch (Exception e) {
                  // Don't!
              }

Naming Conventions

  • Constants should be uppercase with words separated by '_'.
  • Consider case-senstivity when naming types.
    E.g., do not use:
                    
                  public class Bel {
                  }
    Do use: 
                    
                  public class BEL {
                  }

Simplify Logic

Simplify conditional logic, when execution flows throw branches in a mutually exclusive fashion.
E.g., avoid using:
            
              if (condition1) {
                  // some code for condition #1
                  return;
              } else if (condition2) {
                  // some code for condition #2
                  return;
              } else {
                  // default code
                  return;
              }
Do use:
            
              if (condition1) {
                  // some code for condition #1
                  return;
              } else if (condition2) {
                  // some code for condition #2
                  return;
              }
              // default code
              return;
The latter syntax matches intent and allows for things like branch prediction by the virtual machine.

Unnecessary Code

Unnecessary code usually comes as a result of software maturity during phases of refactoring. Here are the types of unnecessary code that should be avoided:
  • Unnecessary null checks
  • Unnecessary else statements
  • Unnecessary casts, instanceof
  • Unused imports
